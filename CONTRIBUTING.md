# Contributing to the Executive Support System

Thank you for your interest in contributing. This is a public project developed in the open, and contributions of all kinds are welcome—feedback, bug reports, documentation improvements, design proposals, and (eventually) code.

---

## Ground Rules

- Be respectful and constructive. Assume good intent.
- This project has a single primary maintainer. Response times may vary.
- All accepted work is the responsibility of the human maintainer, regardless of how it was created.
- AI-assisted contributions are permitted. Please disclose this in your pull request. See [AI_USAGE.md](AI_USAGE.md).

---

## What You Can Contribute

### Right Now (Documentation Phase)

- Corrections or improvements to existing documentation
- Suggestions for the architecture, design, or capture workflow
- Research references relevant to the [docs/research/](docs/research/) folder
- Proposals for Architecture Decision Records (ADRs)
- Issue reports for anything unclear, incorrect, or missing

### Later (Once Prototype Work Begins)

- Notion template improvements
- Automation adapter implementations
- AI enhancement integrations
- Test cases and workflow validations

---

## How to Contribute

### Reporting Issues

Use the GitHub Issue templates:

- **Bug report** — something is wrong, broken, or inconsistent
- **Feature request** — an idea for something new or different
- **Documentation** — corrections, gaps, or unclear content

Include as much context as you can. A clear description of the problem is more useful than a proposed solution, but both are welcome.

### Submitting Changes

1. Fork the repository
2. Create a branch with a descriptive name (`docs/update-capture-ingestion`, `fix/roadmap-typo`)
3. Make your changes
4. Open a pull request using the provided template
5. Describe what you changed and why

For significant changes, please open an issue first to discuss the proposal before investing time in implementation.

### Documentation Standards

- Write in plain English. Avoid jargon where possible.
- Use sentence case for headings.
- Keep lines reasonably short (soft limit: 120 characters).
- Prefer short paragraphs over long walls of text.
- Use "Personal" and "Work" as the default public profile names.
- Do not make diagnostic, treatment, or therapeutic claims.
- Do not assume a specific AI provider, LLM, automation platform, or recorder.

### Architecture Decision Records (ADRs)

Significant architectural decisions should be documented as ADRs in [docs/decisions/](docs/decisions/). Use the template in that folder. Propose new ADRs via pull request.

## AI-Assisted Contributions

AI tools may be used to draft, edit, or review contributions. If AI assistance was used in preparing your pull request, please note it in the PR description. The contributor remains responsible for the accuracy and quality of the submitted content.

---

## Code of Conduct

Be constructive. Be direct. Respect that this is someone's personal productivity tool being built in public, not an enterprise project with dedicated support resources.

There is no formal code of conduct document yet. If one is needed, it will be added.
